# Hybrid write granularity: document builder, fine-grained logging

Workout writes use two deliberately different shapes. The **builder** saves the whole nested workout (exercises + sets) as one atomic document (`workouts.create`/`update`, full child replace in a transaction) and is only legal while the workout is `planned`. **Logging** uses small single-purpose mutations (`start`/`finish`/`cancel`, `completeSet`/`uncompleteSet`/`updateSet`/`addSet`); once a workout is `in_progress`, its exercise structure is frozen — only sets can be edited or appended.

Rationale: the builder is a form, and forms want atomic saves with no partial-failure states; mid-workout logging is a hot path that wants one-round-trip optimistic mutations. Freezing structure at start keeps reconciliation trivial (no merging completed sets against a restructured document) and matches how most workout apps behave. The set-level mutations are also shaped like the events the future realtime sync mode will broadcast, so that fast-follow extends this surface rather than replacing it. Rejected: all-document (resubmitting the workout to check off a set is heavy and racy) and all-fine-grained (the builder becomes a swarm of partial-failure states).
